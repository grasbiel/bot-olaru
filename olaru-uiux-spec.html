Aplicar escala de 4px em toda a UI via --spacing-*. Nunca usar valores arbitrários (ex: 13px, 22px).

Border Radius
  • Botões e inputs: 6px
  • Cards e painéis: 10px
  • Modals: 14px
  • Chips/badges: 4px (não pill)
  • Avatares: 50%
Sombras (Elevação)
  • Level 0: sem sombra (painel flat)
  • Level 1: 0 1px 3px rgba(0,0,0,.4)
  • Level 2: 0 4px 12px rgba(0,0,0,.5)
  • Modal/Dropdown: Level 3 com overlay
Transições
  • Hover states: 150ms ease
  • Fade-in cards: 200ms ease-out
  • Kanban drag: 100ms (rápido)
  • Modals/sheets: 250ms ease-in-out
  • Nunca >400ms em ações de carga

Padrões UX Mobile
  • Touch targets mínimos de 44×44px (WCAG 2.5.5)
  • Botões principais fixos no bottom sheet (alcance do polegar)
  • Swipe-to-action nos cards de visita (ação secundária)
  • Scroll infinito apenas com pull-to-refresh explícito
  • Confirmação obrigatória antes de "Finalizar Visita"
  • Campo de observação + foto com câmera nativa via input[capture]
Estratégia Offline-First
  • Service Worker (PWA) armazena lista de visitas em Cache API
  • Ações (Cheguei / Finalizar) enfileiradas no IndexedDB
  • Indicador visual permanente de status de conexão
  • Sync automático ao reconectar — com toast de confirmação
  • Dados de visita nunca se perdem — persistência local garantida
12.4

Estados de Interface (UI States)

Todo componente que busca dados deve implementar todos os 5 estados abaixo. Nunca renderizar tela vazia sem tratamento.

⏳
Loading (Skeleton)
Skeleton screens em vez de spinners genéricos. Mesma estrutura visual do conteúdo real.
📭
Empty State
Mensagem contextual + ação primária. Ex: "Nenhuma visita hoje. Criar primeira visita →"
💥
Error State
Mensagem amigável, sem jargão técnico + botão "Tentar novamente". Logar erro no console para debug.
✅
Success State
Toast de confirmação com ação de desfazer quando cabível. Duração: 4–5 segundos.
🚫
Disabled State
Opacidade 40%. Cursor not-allowed. Tooltip explicando o motivo (ex: "Sem permissão de gerente").
📶
Offline State (Mobile)
Banner persistente âmbar no topo. Ações locais habilitadas. Dados sincronizados em background.
Regra de Ouro: Nenhuma tela pode exibir uma lista ou tabela vazia sem orientar o usuário sobre o próximo passo. Um estado vazio bem projetado reduz abandono e confusão.
12.5

Acessibilidade (WCAG 2.1 — Nível AA)

🎨
Contraste de Cor
Texto sobre fundo: mínimo 4.5:1 (normal) e 3:1 (grande/bold). Usar ferramenta de contraste no Figma ou browser DevTools. Não transmitir informação exclusivamente por cor — sempre associar ícone ou texto.
⌨️
Navegação por Teclado
Todo elemento interativo deve ser alcançável via Tab. Focus ring visível e com contraste. Kanban deve suportar mover cards com teclado (Space para pegar, arrow keys para mover, Enter para soltar — padrão ARIA).
🏷️
Semântica HTML e ARIA
Usar <button> para ações, <a> para navegação. Adicionar aria-label em ícones sem texto. Kanban: role="list" nas colunas, role="listitem" nos cards, aria-grabbed e aria-dropeffect no drag-and-drop.
🔊
Leitores de Tela
Testar com NVDA (Windows) ou VoiceOver (iOS/macOS) as telas críticas: login, dashboard, Kanban e formulário de visita. Anunciar atualizações em tempo real (SSE/WebSocket) com aria-live="polite".
📱
Touch Targets (Mobile)
Área mínima de toque: 44×44px (Apple HIG) ou 48×48px (Material Design). Espaçamento entre botões adjacentes: mínimo 8px para evitar toques errados, crítico no contexto de campo do técnico.
🌓
Preferências do Sistema
Respeitar prefers-reduced-motion: desabilitar animações para usuários com epilepsia ou sensibilidade vestibular. Implementar via @media query em CSS e no Angular Animation Builder.
12.6

Padrões de Feedback Visual

✅ Visita agendada com sucesso para 30 Mar, 14:00 — Equipe A.
❌ Erro ao salvar: equipe não disponível neste horário. Tente outro turno.
⚠️ Você está prestes a cancelar uma visita confirmada. Esta ação notificará o cliente.
ℹ️ Nova mensagem do lead Silva & Cia no Chatwoot. Abrir conversa →
Toasts e Notificações
  • Posição: bottom-right no desktop, top-center no mobile
  • Duração: 4s (info), 5s (sucesso), persistente (erro crítico)
  • Máximo 3 toasts simultâneos — empilhar verticalmente
  • Botão X sempre visível para dispensar manualmente
  • Ação de Undo para operações destrutivas (5s)
Confirmações Destrutivas
  • Modal de confirmação para: cancelar visita, excluir máquina, inativar equipe
  • Botão destrutivo: vermelho, à direita, não é o padrão/primeiro foco
  • Texto do botão: verbo da ação — "Cancelar Visita" e não apenas "Confirmar"
  • Double-confirmation (digitar nome) para ações irreversíveis de alta criticidade

12.8

Performance Percebida (UX de Velocidade)

Princípio: O app não precisa ser rápido — precisa parecer rápido. Técnicas de perceived performance são tão importantes quanto otimizações reais.
Otimistic UI
  • Atualizar o Kanban antes da confirmação do servidor
  • Reverter automaticamente em caso de erro da API
  • Aplicar em: trocar status, confirmar visita, check-in técnico
Lazy Loading Angular
  • Módulo de relatórios: carregado apenas ao acessar a rota
  • Módulo de configurações do robô: lazy (uso raro)
  • Imagens de fotos de visita: loading="lazy"
  • Bundle inicial < 300KB (gzip)
Core Web Vitals
  • LCP (Largest Contentful Paint): < 2.5s
  • FID / INP: < 100ms
  • CLS (Layout Shift): < 0.1
  • Medir com Lighthouse CI no pipeline de deploy
